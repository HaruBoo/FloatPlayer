Swift基礎編: 言語機能とAppKitの土台

FloatPlayerで実際に使ったSwiftの言語機能と、SwiftUI/AppKitを橋渡しする仕組みを、コードと対応させながら解説します。

0. Swiftの基礎知識(ざっくり)

本編に入る前に、このアプリのコードを読むうえで最低限知っておくと楽になるSwiftの基礎を、 ほんとに軽くだけおさらいします。すでに知っている場合は1章まで読み飛ばして構いません。

let と var

letは再代入できない定数、varは再代入できる変数を宣言します。 Swiftでは「変わらない値にはletを使う」のが強く推奨されるスタイルで、 このプロジェクトでもvarは「本当に変化する状態」(例:@Published var mode)にだけ使っています。 

let apiURL = "https://www.googleapis.com/youtube/v3/videos" // 再代入不可
var isLoading = false                                       // 再代入可能
isLoading = true

型推論とOptional(nilを許す型)

Swiftはlet name = "FloatPlayer"のように書くだけで型(この場合String)を推論してくれます。 型注釈を省略できますが、型自体は静的に決まっている(後から別の型を代入することはできない)強い型付け言語です。

Swiftの大きな特徴がOptional。「値があるかもしれないし、無い(nil)かもしれない」ことを 型のレベルで表現する仕組みで、型名の後ろに?を付けます。 

var videoURL: URL?          // URLが入っているか、nilかのどちらか
var currentVideoID: String? // 動画未選択の状態をnilで表現している

Optionalの中身を安全に取り出すにはif letやguard letを使います。 

guard let id = Self.extractYouTubeID(from: youtubeInput) else { return }
// ここから先では id は String?ではなく、中身のあるString として使える

guard ... else { return }は「条件を満たさなければここで処理を打ち切る」という書き方で、 ネストを深くせずに「早期リターン」できるためSwiftでは非常によく使われます。

関数・メソッド・引数(前提知識)

ここから先で「関数」「メソッド」「引数」という言葉が繰り返し出てくるので、先に整理しておきます。 関数は「名前を付けてまとめた、ひとまとまりの処理」のことです。引数はその処理に渡す 材料(入力)で、関数は必要に応じて結果を戻り値として返します。メソッドは 「structやclassの中に定義された関数」を指す呼び方で、やっていることは関数と同じです。 

// 「2つの数を渡すと、足し算した結果を返す関数」
func add(_ a: Int, _ b: Int) -> Int {
    return a + b
}
let result = add(1, 2) // resultは3

クロージャ(無名関数)

クロージャは{ 引数 in 処理 }という形の「名前を付けずにその場で作る関数」です。他の言語の ラムダ式・アロー関数に相当します。「この処理を後で実行してほしい」というコールバック (何かが起きたタイミングで呼び返してもらう関数)として、非同期処理(結果が返ってくるまで プログラムの他の部分を止めずに進める書き方)の中で頻繁に登場します。 

URLSession.shared.dataTask(with: url) { data, _, _ in
    // このブロック自体がクロージャ。dataが返ってきたときに後から呼ばれる
    print(data ?? "no data")
}.resume()

クロージャが後から実行される(今すぐではない)関数の引数になっているとき、Swiftはその引数に @escapingという注釈を要求することがあります。「この関数を抜けた後もクロージャが生き残る」 ことを明示するための注釈です。

struct と class の違い

Swiftには値型のstructと参照型のclassの両方があります。 値型は「コピーされて渡される」、参照型は「同じ実体(メモリ上の同じ場所にある本体)への 参照(道しるべのようなもの)が共有される」というイメージです。たとえば2枚の紙にそれぞれ内容を コピーする(struct)のか、1枚の紙を2人で見に行く(class)のか、という違いです。片方を書き換えたときに、 もう片方まで一緒に変わってほしいならclass、変わってほしくないならstructを選びます。 SwiftUIのビュー(ContentView)やChapterのようなデータの入れ物はstruct、 状態を持ち続けてUI全体で共有したいMediaViewModelはclass、という使い分けをしています。 

struct Chapter: Identifiable {   // 値型: コピーされる、単純なデータ
    let id = UUID()
    let seconds: Int
    let title: String
}

final class MediaViewModel: ObservableObject {  // 参照型: 複数箇所で同じインスタンスを共有
    @Published var mode: PlaybackMode = .youtube
}

enum(列挙型)

再生モードのように「決まった選択肢のどれか」を表すのにenum(列挙型)を使います。 「YouTubeか、写真か、動画か、そのどれか1つ」のような、あらかじめ決まったパターンしか 存在しない値を表すのに向いています。Swiftのenumは他言語と違い、計算プロパティ (呼ばれるたびに計算して値を返す、変数のように見えるもの)やメソッドを持てるのが特徴です。 

enum PlaybackMode: String, CaseIterable, Identifiable {
    case youtube, photo, video

    var label: String {
        switch self {
        case .youtube: return "YouTube"
        case .photo: return "写真"
        case .video: return "動画"
        }
    }
}

プロトコル(protocol)

プロトコルは「このメソッド/プロパティを持っていること」という約束事(ルール)を定義するもので、 他言語のインターフェースに近い概念です。「このルールに従いますと宣言したクラスは、 約束したメソッドを必ず実装しなければならない」という仕組みだと考えると分かりやすいです。 AppDelegateがNSApplicationDelegateというプロトコルに 準拠する(: NSObject, NSApplicationDelegateと書いて「このルールに従います」と宣言する)ことで、 AppKit(macOSのアプリの土台となる部品集)側からapplicationDidFinishLaunching(アプリの起動が 完了した時に呼ばれるメソッド)などの決まったタイミングで、自動的に呼び出してもらえるようになります。 

final class AppDelegate: NSObject, NSApplicationDelegate {
    func applicationDidFinishLaunching(_ notification: Notification) {
        // アプリ起動完了時にAppKitから呼ばれる
    }
}
Tips: ここまでの基礎が分かれば読み進められる

let/var、Optional、クロージャ、struct/class、enum、protocolの6つが分かれば、 このアプリのコードの9割は読めるようになります。ここから先は、それらの基礎知識を前提に、 このアプリならではの実践的な使い方を見ていきます。

1. @MainActor とアクター分離

まず前提として、アプリは1つの処理を順番にこなしているわけではなく、内部に複数の スレッド(処理の流れ)を持てます。たとえば「画面を描画する」処理と「インターネットから データをダウンロードする」処理を同時に進められると、ダウンロード中も画面が固まらず快適に 操作できます。ただし複数のスレッドが同じデータを同時に書き換えようとすると、 書き込みのタイミングによって結果が変わってしまうデータ競合(race condition)という 厄介なバグが起こります。しかも再現性が低く、たまにしか起きないため見つけにくいのが特徴です。

さらにmacOS/iOSのAppKit/SwiftUIには「ウィンドウやボタンなどの画面部品は、必ずメインスレッド (アプリの中心となる、画面表示を担当する特別なスレッド)からしか操作してはいけない」という 制約があります。これを破ると、クラッシュしたり、表示が崩れたりします。 Swift 6では、この手のデータ競合を実行前(コンパイル時)に検出する「厳格な並行性チェック」が 既定で有効になっており、Swiftはこの制約を@MainActorという属性(型や関数に付ける追加情報)で 型システムに組み込んでいます。 

@MainActor
final class AppDelegate: NSObject, NSApplicationDelegate {
    private var panel: NSPanel!
    // ...
}

クラス全体に@MainActorを付けると、そのクラスのすべてのプロパティ・メソッドが 「メインスレッド上でのみアクセス可能」という制約を持つようになります。これにより、 バックグラウンドスレッドから誤ってpanelを触ってしまうようなコードは、 実行して初めて気づくのではなくコンパイルエラーとしてその場で教えてもらえます。 「実行するまで分からなかったバグを、書いている最中に指摘してもらえる」というのが この仕組みの実用上のメリットです。

実際に踏んだエラー: main.swiftトップレベルの分離違反

main.swiftという名前のファイルは特別で、その中にクラスや関数の定義を書かず 直接命令を並べると、そこがアプリの「エントリーポイント」(実行が最初に始まる場所)になります。 このトップレベル(クラスや関数の外側にある部分)は、実行時には確かにメインスレッド上で 動いているにもかかわらず、Swiftの型システム上は@MainActorに分離されているとは 見なされないという食い違いがあります。「実際には安全なのに、コンパイラはそれを 証明できない」という状態です。そのため、@MainActor指定したAppDelegate()を トップレベルから素朴に呼ぶと、次のコンパイルエラーになりました。 

error: call to main actor-isolated initializer 'init()'
in a synchronous nonisolated context

このエラーメッセージは「メインアクターに分離された初期化処理を、そうでない(nonisolatedな) 同期的な文脈から呼び出そうとしている」という意味です。エラー文をそのまま読めば 原因がつかめる、というのは実務でもよくあることなので、エラーメッセージの単語を1つずつ 分解して読む癖をつけておくと役立ちます。

解決策はMainActor.assumeIsolatedという関数を使い、「ここは型システム上は 証明できないが、実際にはメインスレッドである」と自分の責任で明示的に橋渡しすることです。 

import AppKit

let app = NSApplication.shared
// main.swiftのトップレベルはmain actor分離されていないため、明示的に橋渡しする
let delegate = MainActor.assumeIsolated { AppDelegate() }
app.delegate = delegate
app.setActivationPolicy(.regular)
app.run()

同じような「call to main actor-isolated ... in a synchronous nonisolated context」という エラーに出会ったら、まず「本当にメインスレッドで実行されている場所かどうか」を確認し、 そうであればMainActor.assumeIsolated、そうでなければ非同期(async)な 呼び出し方に直す、という2択で考えると解決の糸口になります。

2. Combineによる状態管理(@Published)

CombineはAppleが提供する、「値の変化を監視して、変わるたびに自動で反応する」ための 部品集です。「ある値を見張っておいて、変わったら知らせてください」という考え方は Observer(監視者)パターンと呼ばれ、Web開発のイベントリスナーや、他言語のRxなどの リアクティブプログラミングと同じ発想です。この仕組みを覚えておくと、他の言語・フレームワークでも 応用が効きます。

FloatPlayerではSwiftUIの画面とAppKitのウィンドウ・メニューバーという「2つのUI」が 同じ状態を参照する必要があります(例: 透明度スライダーの値は、SwiftUIのSliderにも AppKitのNSMenuItemにも影響します)。この2つを同期させる橋渡しに、Combineの @Publishedとsinkを使っています。 

@MainActor
final class MediaViewModel: ObservableObject {
    @Published var mode: PlaybackMode = .youtube
    @Published var isUIHidden: Bool = false
    @Published var mediaOpacity: Double = 1.0
    // ...
}

ObservableObject(「監視できるオブジェクトです」と宣言するプロトコル)に準拠した クラスの@Publishedプロパティが変化すると、SwiftUI側は自動的に再描画されます。 それとは別に、AppKit側(AppDelegate)ではsink(「この値の変化を受け取って 処理する」という意味の購読メソッド)で変化を購読し、ステータスバーのメニュー項目など AppKitのAPIを手動で更新します。SwiftUI側は「自動で再描画される」、AppKit側は 「sinkで自分から変化を拾いにいく」という非対称な関係になっているのがポイントです。 

viewModel.$isUIHidden
    .receive(on: RunLoop.main)
    .sink { [weak self] hidden in
        self?.uiHiddenMenuItem.state = hidden ? .on : .off
        self?.uiHiddenMenuItem.title = hidden ? "UIを表示" : "UIを隠す"
    }
    .store(in: &cancellables)

ここで使われている[weak self]を理解するには、Swiftのメモリ管理の仕組みを先に 知っておく必要があります。SwiftはARC(Automatic Reference Counting)という仕組みで メモリを管理していて、「あるオブジェクトを何か所から参照されているか」を数え、 参照する側が誰もいなくなった時点でそのオブジェクトをメモリから解放します。ここで self(このクロージャを書いているクラス自身のインスタンス)を、クロージャの中で そのまま(強参照で)使うと、次のような輪ができてしまいます。

  • self(このクラスのインスタンス)がcancellablesという集合を持っている
  • cancellablesがsinkのクロージャを持っている
  • そのクロージャがselfを(強参照で)持っている

これは循環参照と呼ばれる状態で、お互いがお互いを必要としているように見えるため、 本当は誰にも使われていなくても参照カウントがゼロにならず、メモリリーク (使われなくなったのに解放されずメモリを圧迫し続けるバグ)の原因になります。 [weak self]は「このクロージャはselfを弱参照で持つ(参照カウントには 数えない)」という指定で、この輪を断ち切ります。弱参照で持ったselfは いつnilになっているか分からないため、クロージャの中ではself?.fooのように Optional(0章参照)として安全に扱います。

「クロージャの中で自分自身(self)を参照するときは、まず[weak self]が 必要かどうかを疑う」という習慣を付けておくと、循環参照によるメモリリークの多くは防げます。

Tips: PassthroughSubjectで「命令」を流す

チャプター選択時のシーク処理には、状態を表す@Publishedではなく PassthroughSubject<Int, Never>を使っています。 

let seekSubject = PassthroughSubject()

func seek(toChapter chapter: Chapter) {
    seekSubject.send(chapter.seconds)
}

@Publishedは「現在の状態」を表すのに向いていますが、シークは「一回きりのイベント」であって 状態ではありません(同じ秒数に2回シークしたいこともあります)。イベント通知には値を保持しない PassthroughSubjectの方が適しています。

3. SwiftUIとAppKitの橋渡し

SwiftUIは「画面の見た目や状態が変わったら自動で再描画する」ことに特化した、比較的新しい 部品集です。一方AppKitは、ウィンドウそのものの細かい挙動(タイトルバーを消す、常に最前面に 浮かせる、など)を制御できる、macOSの土台となる古くからの部品集です。FloatPlayerは 「ウィンドウの挙動はAppKitで細かく制御したいが、画面の中身はSwiftUIで手早く書きたい」 という理由から、この2つを両方使っています。SwiftUIとAppKitはそれぞれ別の仕組みなので、 相互に行き来するための「橋渡し役」が必要になります。

NSHostingViewでSwiftUIをAppKitウィンドウに載せる

FloatPlayerはWindowGroup(SwiftUIだけで完結する、通常のSwiftUI Appのひな形)のような 通常のSwiftUI Appライフサイクル(アプリが起動してから終了するまでの一連の流れ)を使わず、 NSPanel(AppKitのウィンドウ)を直接作って、そのcontentView(ウィンドウの中身を 表すプロパティ)にSwiftUIビューを載せています。この橋渡し役がNSHostingViewです。 

let hosting = NSHostingView(rootView: ContentView(viewModel: viewModel))
newPanel.contentView = hosting

これにより、「タイトルバーを消す」「常に最前面に浮かせる」「クリックスルーさせる」といった AppKit特有の細かいウィンドウ制御をしながら、画面の中身はSwiftUIで書けるようになります。 (このNSHostingViewには重大な落とし穴があり、詳しくはNSWindow/NSPanel編で扱います。)

NSViewRepresentableでAppKitの部品をSwiftUIに載せる

逆方向の橋渡しがNSViewRepresentableです。YouTube再生にはWKWebView(AppKit/WebKitのクラス)が 必要ですが、SwiftUIには標準のWebView部品がないため、自分でラップしています。 

struct YouTubeWebView: NSViewRepresentable {
    @ObservedObject var viewModel: MediaViewModel

    func makeNSView(context: Context) -> WKWebView {
        let webView = WKWebView(frame: .zero, configuration: config)
        // ...
        return webView
    }

    func updateNSView(_ webView: WKWebView, context: Context) {
        // SwiftUIの状態が変わるたびに呼ばれる
    }

    static func dismantleNSView(_ webView: WKWebView, coordinator: Coordinator) {
        // このビューがSwiftUIツリーから外れるときに呼ばれる
        webView.pauseAllMediaPlayback(completionHandler: nil)
    }
}

makeNSView(初回生成)、updateNSView(SwiftUIの再描画のたびに呼ばれる)、 dismantleNSView(ビューが破棄されるとき)という3つのライフサイクルメソッドを実装するのが基本形です。

ハンズオン: Coordinatorパターンで外部からの命令を受け取る

「チャプターを選んだら、再読み込みせずに今再生中のWKWebViewをシークする」ためには、 SwiftUIの外(メニューバーのクリック)からWKWebViewに命令を送る経路が必要になります。 NSViewRepresentableはCoordinatorという補助オブジェクトを持てるので、 そこでCombineの購読を持たせています。 

final class Coordinator {
    private weak var webView: WKWebView?
    private var cancellables = Set()

    func attach(_ webView: WKWebView) {
        self.webView = webView
        viewModel.seekSubject
            .sink { [weak self] seconds in
                self?.run("fpSeek(\(seconds));")
            }
            .store(in: &cancellables)
    }
}

こうしておくと、AppDelegate側は「seekSubjectに値を流すだけ」で、実際にWKWebViewの JavaScriptを実行する責務はCoordinatorに閉じ込められます。SwiftUIの外の世界(メニューバー)と 中の世界(WKWebView)を、CombineのSubjectを介して疎結合(お互いの内部実装を知らなくても やり取りできる、依存の少ない結びつき方)につなぐパターンです。「AがBの中身を直接いじる」のではなく 「Aは合図を送るだけ、Bがそれを受けて自分の判断で動く」という形にしておくと、後からAやBの 実装を個別に変更しやすくなります。

4. DispatchSourceによるファイル監視

「新しいスクリーンショットが増えたら検知したい」という機能を作るとき、まず思いつくのは ポーリング(一定間隔でフォルダの中身を確認しにいく方式)です。実装は簡単ですが、 間隔を短くすると無駄にCPUを使い続け、長くすると検知が遅れるというトレードオフがあります。 FloatPlayerでは代わりに、OS(macOSの内部で常に動いているカーネルという中核部分)が 「ファイルが変化したら教えてあげる」と直接通知してくれるkqueueという仕組みを、 DispatchSourceというSwiftの部品経由で使っています。「こちらから毎回聞きに行く」のではなく 「向こうから知らせてもらう」ため、CPUを無駄遣いせず、かつ検知も速いという方式です。 

directoryFD = open(dir.path, O_EVTONLY)
let source = DispatchSource.makeFileSystemObjectSource(
    fileDescriptor: directoryFD,
    eventMask: .write,
    queue: .main
)
source.setEventHandler { [weak self] in
    self?.checkForNewFiles(in: dir)
}
source.resume()

open(path, O_EVTONLY)は、監視したいフォルダをファイルディスクリプタ (OSがファイルやフォルダを識別するために割り振る、番号のようなもの)として開く処理です。 O_EVTONLYは「読み書きはしないが、変化だけ監視したい」ときに使うフラグ(オプションの指定)です。 こうして得たファイルディスクリプタに対して.writeイベントを監視すると、 そのディレクトリの中身が変化した瞬間(ファイルの追加・削除・リネーム)にハンドラ (イベントが起きたときに呼ばれる処理)が呼ばれます。これはPythonのwatchdogや Node.jsのfs.watchと同じ、OSネイティブ(その言語専用ではなく、OS自体が提供する)の ファイル監視APIです。同じ発想の仕組みは他の言語にもあるので、「ポーリングではなく OSに通知してもらう」という考え方自体を覚えておくと、言語が変わっても応用できます。

注意: 差分検出はアプリ側で行う必要がある

.writeイベントは「何かが変わった」としか教えてくれません。「何が追加されたか」は 自分でディレクトリの中身を列挙して、前回保持していたファイル名の集合(Set<String>)と 差分(subtracting)を取る必要があります。 

let current = Self.listImageFiles(in: dir)
let newNames = current.subtracting(knownFiles)
knownFiles = current
実際に踏んだエラー: イベント発火直後は書き込みが終わっていないことがある

.writeイベントは「書き込みが完了した後」ではなく「書き込みが始まった/進行した」タイミングでも 発火することがあります。スクリーンショットのように大きな画像ファイルだと、イベントを受けた直後に NSImage(contentsOf:)で読み込もうとすると、まだファイルが書き終わっておらずデコードに失敗することがありました。 

// 書き込みが完了しきっていないタイミングで読むと失敗するため少し待つ
DispatchQueue.main.asyncAfter(deadline: .now() + 0.3) { [weak self] in
    self?.onNewScreenshot(newest)
}

根本的な解決(ファイルサイズが安定するまで待つ、など)もできますが、今回は「0.3秒待ってから読む」という 簡易的なデバウンス(短時間に連続して起きるイベントを間引き、少し待ってから1回だけ 処理する手法)で十分実用的な精度が出た。厳密な正しさより「実用上困らない簡易な対処」を 選ぶのも、実際の開発ではよくある判断です。

実際に踏んだバグ: 隠しファイル名で一瞬だけ現れてリネームされる

スクリーンショットは書き込み中、.で始まる一時的な隠しファイル名(例: .png)で先に現れてから、最終的なファイル名にリネームされることがあります。 差分検出がこの隠しファイルを「新しいファイル」として拾ってしまうと、0.3秒後に読みにいった時点では 既にリネームされていてパスが存在せず、読み込みに失敗します。 

// スクリーンショットは書き込み中、"."で始まる一時的な隠しファイル名で
// 現れてから最終的なファイル名にリネームされることがあるため対象から除く
return Set(names.filter {
    !$0.hasPrefix(".") && imageExtensions.contains(($0 as NSString).pathExtension.lowercased())
})

これは、ファイルが書き込まれるタイミングとアプリが読みにいくタイミングのズレが 原因で起きる、一種の競合状態(1章で説明したデータ競合と同じ発想の問題)です。 隠しファイルを最初から一覧に含めないようにするだけで、最終的な(リネーム後の)ファイル名だけが 「新しいファイル」として検出されるようになり、このタイミング問題そのものが起きなくなります。

4.5. 機能: クリップボードのみのスクリーンショットを拾う

Cmd+Ctrl+Shift+4のような「ファイル保存を伴わないスクリーンショット」は、上記のフォルダ監視では 検知できません。4章では「ポーリングより、OSに通知してもらう方が良い」と 説明しましたが、macOSのクリップボード(NSPasteboard)には、中身が変わったことを 知らせてくれる通知の仕組みが公開されていません。このようにOSが通知の手段を用意していない 場合は、ポーリングが現実的な選択肢になります。そこでClipboardImageWatcherは NSPasteboard.general.changeCount(クリップボードの中身が変わるたびに増える番号)を 0.5秒間隔でポーリングし、前回と値が変わっていたら「クリップボードの中身が変わった」とみなします。 

timer = Timer.scheduledTimer(withTimeInterval: 0.5, repeats: true) { [weak self] _ in
    Task { @MainActor in self?.checkClipboard() }
}

private func checkClipboard() {
    let pasteboard = NSPasteboard.general
    guard pasteboard.changeCount != lastChangeCount else { return }
    lastChangeCount = pasteboard.changeCount
    guard let image = NSImage(pasteboard: pasteboard) else { return }
    onNewImage(image)
}

macOSには「クリップボードの中身がスクリーンショット由来か」を判別する公開APIが無いため、 この仕組みは新しい画像がコピーされた時点で全て拾います。通常の画像コピー操作にも反応してしまうため、 既定はオフにして、メニューバーから明示的に有効化する追加機能として提供しています。

実際に踏んだバグ: ファイルをコピーしただけなのに画像が浮く

Finderでファイル(画像でなくても)をコピーすると、NSPasteboardにはファイルURLと同時に、 そのファイルのアイコン画像がTIFF形式(PNGやJPEGと同じ、画像データの保存形式の1つ)で 一緒に乗ることがあります。NSImage(pasteboard:)は 「画像として解釈できるものが乗っているか」しか見ないため、このアイコンを「新しくコピーされた画像」と 誤認してフローティング表示してしまっていました。 

// Finderでファイルをコピーした時、そのファイルのアイコン画像がNSImage(pasteboard:)で
// 拾われてフローティング表示されてしまうことがあるため、ファイルURLがある場合は
// (通常のファイルコピーとみなして)画像コピーとして扱わない
let hasFileURL = !(pasteboard.readObjects(forClasses: [NSURL.self], options: nil) ?? []).isEmpty
guard !hasFileURL, let image = NSImage(pasteboard: pasteboard) else { return }
onNewImage(image)

「画像かどうか」ではなく「ファイル参照が乗っているかどうか」を先に確認することで、ファイルコピーと 画像コピーを区別できます。同じ理由で、明示的な「貼り付け」ボタン側(pastePhotoFromClipboard())にも 同様の対策を入れ、ファイルURLがあって、かつそれが画像ファイルであれば、アイコンではなく実体のファイルを 読み込むようにしています。

5. 機能: 透明度スライダーを「映像用」と「UI用」に分ける

FloatPlayerには透明度スライダーが2つあります。mediaOpacityはYouTube/写真/動画そのものの透明度、 uiOpacityは上下バーやスライダーなどUI部品の透明度で、それぞれ独立に0.15〜1.0の範囲で動かせます。 

// MediaViewModel.swift
@Published var mediaOpacity: Double = 1.0 // YouTube/写真/動画の透明度
@Published var uiOpacity: Double = 1.0    // 上下バーなどUIの透明度

実装としては、SwiftUIの.opacity()修飾子を「どの階層のViewに付けるか」を厳密に分けるだけです。 上下バー全体にはuiOpacityを、実際に映る中身(YouTubeWebView・Image・ VideoPlayer)にだけmediaOpacityを付けています。 

// ContentView.swift
topBar
    .background(DragHandle())
    .opacity(viewModel.uiOpacity)
// ...
YouTubeWebView(viewModel: viewModel)
    .opacity(viewModel.mediaOpacity)
実際に踏んだバグ: 説明文や入力欄まで透明になってしまう

最初の実装では、YouTubeの入力欄やヒントテキストを含むセクション全体に mediaOpacityを1つだけ付けていました。これだと映像を透明にしようとスライダーを動かすたびに、 TextField(URL入力欄)や説明文まで一緒に薄くなり、「文字が読めない」「入力欄がどこにあるか 分からない」状態になってしまいました。 

// NG: セクション全体に1つのopacityを付けていた
youtubeSection
    .opacity(viewModel.mediaOpacity)

原因はシンプルで、「透明にしたいのは映像だけ」という意図に対して、.opacity()を 付ける階層が広すぎたことでした。修正は、入力欄やヒントテキストを囲む親Viewから.opacity()を外し、 実際の映像コンポーネント(YouTubeWebView・Image・VideoPlayer)にだけ 個別に付け直すことでした。 

// OK: 入力欄・説明文には付けず、映像そのものにだけ付ける
if !viewModel.isUIHidden {
    TextField("YouTubeのURLまたは動画ID", text: $viewModel.youtubeInput)
    // opacityなし。常にはっきり読める
}
if viewModel.currentVideoID != nil {
    YouTubeWebView(viewModel: viewModel)
        .opacity(viewModel.mediaOpacity) // 透明にしたいのはここだけ
}

SwiftUIの画面は、部品(Viewと呼ばれる)を入れ子(親の中に子、その中にまた子…)にして 組み立てる木構造(ツリー)になっています。.opacity()のような修飾子は、 それを付けた部品と、その内側にあるすべての子部品に効きます。つまり「何を透明にしたいか」を 正確にコントロールするには、opacityを親の広い階層に付けるのではなく、本当に透明にしたい 部品(この場合は映像そのもの)にだけピンポイントで付ける必要がある、という教訓を得ました。 「広い範囲に効果をかけたら意図しないものまで巻き込んだ、対象を絞り込んだら直った」という パターンは、SwiftUI以外のUIフレームワーク(CSSなど)でもよく起きるので覚えておくと役立ちます。
