NSWindow/NSPanel編: 消えるウィンドウと巨大化するウィンドウ

「常に最前面に浮くウィンドウ」を作る過程で実際に踏んだ、AppKitのウィンドウ管理まわりの重いバグ4つを解説します。

1. NSPanelの基本設定

FloatPlayerのメインウィンドウは、通常のNSWindowではなくNSPanelで作っている。 NSPanelはNSWindowのサブクラスで、パレット・ツールウィンドウ向けの挙動(後述の nonactivatingPanelなど)を追加で持つ。 

let newPanel = NSPanel(
    contentRect: NSRect(x: 200, y: 200, width: 420, height: 280),
    styleMask: [.titled, .resizable, .closable, .miniaturizable,
                .nonactivatingPanel, .fullSizeContentView],
    backing: .buffered,
    defer: false
)
newPanel.titleVisibility = .hidden
newPanel.titlebarAppearsTransparent = true
newPanel.level = .floating
newPanel.isOpaque = false
newPanel.backgroundColor = .clear
設定意味
.nonactivatingPanelクリックしても他のアプリのアクティブ状態を奪わない
.fullSizeContentViewコンテンツ領域がタイトルバーの裏側まで広がる(独自デザインのタイトルバー風UIを作るときの定番)
level = .floating通常ウィンドウより手前のレイヤーに常に表示する
isOpaque = false / backgroundColor = .clearウィンドウの背景を透明にし、SwiftUI側で自由に描画する
注意: .closable/.miniaturizableを付けないと信号機ボタンが「見えるだけ」になる

.titledを指定すると赤・黄・緑の信号機ボタンは常に描画される。しかし .closable・.miniaturizableをstyleMaskに含めていないと、 ボタンは表示されるが無効化(グレーアウト)された状態になる。「緑(拡大)だけ効くのに赤と黄が反応しない」 という症状が出たら、まずこのstyleMaskの組み合わせを疑うとよい。

2. バグ1: 入力しようとするとウィンドウが消える(hidesOnDeactivate)

NSPanelにはhidesOnDeactivateというプロパティがあり、 既定値がtrueになっている。これは「自分のアプリが非アクティブになったら、 このパネルを自動的に隠す」という設定。

FloatPlayerはDockに出ないアクセサリ型アプリとして作っていたため、他のアプリ(ターミナルなど)に 一瞬フォーカスが移っただけで、このパネルが勝手に非表示になってしまっていた。ユーザーからは 「テキストフィールドをクリックして入力しようとすると、ウィンドウごと消える」ように見える。 

// NSPanelは既定でhidesOnDeactivate=trueのため、他アプリにフォーカスが移ると
// パネルごと隠れてしまう。常時フロートさせたいのでfalseにする
newPanel.hidesOnDeactivate = false

3. バグ2: システムがアプリを勝手に終了させる(Automatic/Sudden Termination)

hidesOnDeactivateを直しても、まだ「操作しようとすると消える」現象が残っていた。 今度はConsole.appの統合ログ(log show、詳しくはデバッグ技法編)で 実際に何が起きているかを確認したところ、次のログが見つかった。 

[com.apple.AppKit:Application] terminate:
[com.apple.AppKit:AutomaticTermination] Terminating
[com.apple.AppKit:Application] Attempting sudden termination (1st attempt)

macOSには「しばらく使われていないアプリを、ユーザーの許可を得ずに自動終了する」仕組みが Lion(10.7)以降存在する(Automatic Termination / Sudden Termination)。 ウィンドウが少ない・アイドル状態のアプリはこの対象になりやすく、常駐ユーティリティであるFloatPlayerは まさにこの挙動に巻き込まれていた。 

func applicationDidFinishLaunching(_ notification: Notification) {
    // macOSの自動終了/サドンターミネーション機構がこのアプリを
    // アイドルな裏方アプリと誤認して勝手にterminate:するのを防ぐ
    ProcessInfo.processInfo.disableAutomaticTermination("常駐して浮遊表示を続ける必要があるため")
    ProcessInfo.processInfo.disableSuddenTermination()
    // ...
}
実際に踏んだエラー: 自分のデバッグコードが引き金になった別のクラッシュ

この調査中、「何が終了の引き金になったか」を特定するためにapplicationShouldTerminate内で NSApp.currentEventの内容をログ出力するデバッグコードを一時的に追加した。 

// 危険: currentEventがキーボードイベントとは限らない
print(event.charactersIgnoringModifiers ?? "nil")

ところがこのイベントはキーボードイベントとは限らず(実際はKitDefinedという種類のシステムイベントだった)、 .charactersIgnoringModifiersをキーボード以外のイベント種別に対して呼び出すとAppKit内部で NSInternalInconsistencyExceptionが発生して落ちる。デバッグのために足したログ自体が新たなクラッシュ原因になる という、実務でもよくある教訓を得た。イベントのtypeを確認してから安全にプロパティへアクセスするよう修正した。 

if event.type == .keyDown || event.type == .keyUp {
    chars = event.charactersIgnoringModifiers ?? "nil"
} else {
    chars = "n/a"
}

4. バグ3: 最後のウィンドウを閉じるとアプリごと終了する

最終的に、上記の調査ログから本当の引き金が判明した。デリゲートメソッド applicationShouldTerminateAfterLastWindowClosedがtrueを返していたため、 何らかの拍子でパネルが「閉じた」扱いになった瞬間、AppKitの _scheduleCheckForTerminateAfterLastWindowClosedという内部処理がタイマー経由でアプリ全体を終了させていた。 

func applicationShouldTerminateAfterLastWindowClosed(_ sender: NSApplication) -> Bool {
    // このアプリはメニューバー常駐のユーティリティなので、
    // パネルが(一瞬でも)閉じた扱いになってもアプリ全体は終了させない。
    // 終了はメニューバーの「終了」からのみ行う。
    false
}

メニューバー常駐アプリ(ステータスバーアイコンから操作するタイプ)では、 「メインウィンドウを閉じる」と「アプリを終了する」を分離するのが定石。この1行を直すことで、 パネルを閉じてもアプリはメニューバーに残り、「パネルを表示」でいつでも復帰できるようになった。

5. バグ4(最大の難所): ウィンドウが画面いっぱいまで巨大化する

UIを整えている途中で、ウィンドウが指定した420×280を大きく超えて、 縦に1000ptを超えるサイズまで膨れ上がる現象が発生した。SwiftUI側のレイアウト(Spacerや .background()など)を1つずつ疑って除去しても直らず、原因の特定に非常に時間がかかった。

ハンズオン: 二分探索でバグを追い詰める

最終的に有効だったのは、SwiftUIのコンテンツをText("Hello")という最小の例に 差し替えて動作を見る、という「二分探索」的アプローチだった。 

let hosting = NSHostingView(rootView: Text("Hello TEMP DEBUG"))

このとき、ウィンドウは指定した420×280ではなく、Textの内容にぴったり合った 119×48という小さいサイズになった。「複雑なUIだと巨大化し、単純なUIだと逆に縮む」—— この2つの実験結果から、SwiftUIコンテンツの理想サイズ(fitting size)がウィンドウそのものを 書き換えているという仮説にたどり着いた。

実際、NSHostingViewは既定でsizingOptionsというプロパティを持っており、 SwiftUI側のレイアウト結果(intrinsic content size)を自分の親ウィンドウのサイズにまで伝播させる。 通常のSwiftUI Appでは「ウィンドウの初期サイズをコンテンツに合わせたい」場面で便利な機能だが、 今回のように「ウィンドウサイズは自分たちで完全に管理したい」場合には邪魔になる。 

let hosting = NSHostingView(rootView: ContentView(viewModel: viewModel))
// NSHostingViewは既定でSwiftUI側の理想サイズ(fittingSize)に合わせて
// ウィンドウ自体をリサイズしてしまう。ウィンドウサイズは自分たちで管理したいので無効化する
hosting.sizingOptions = []

この1行を追加した瞬間、何十行ものレイアウト修正を試しても直らなかった巨大化が完全に解消した。

Tips: frameの実測ログで「思い込み」を排除する

このバグ調査で決定的だったのは、スクリーンショットの目視ではなく、 DispatchQueue.main.asyncAfterで複数のタイミングにおけるpanel.frameと hosting.fittingSizeを実際にprintしたこと。 

for delay in [0.0, 0.5, 1.0, 2.0, 5.0] {
    DispatchQueue.main.asyncAfter(deadline: .now() + delay) {
        print("t=\(delay) window.frame=\(newPanel.frame) " +
              "hosting.fittingSize=\(hosting.fittingSize)")
    }
}

スクリーンショットの目測は「他のウィンドウとの重なり」「Retinaのスケール換算」などでいくらでも 誤読が起きる。数値を直接ログに出すことで、「本当にウィンドウのframeが変わっているのか」 「それとも見た目だけの問題か」を一発で切り分けられた。

6. ウィンドウレベルを状況に応じて切り替える

「常に最前面」は便利だが、他のアプリを操作しているときまでFloatPlayerが手前に居座るのは邪魔になる。 そこでNSWorkspaceのアプリ切り替え通知を購読し、自分がアクティブな間だけ最前面、 それ以外は通常レベルに切り替えるようにした。 

NSWorkspace.shared.notificationCenter.addObserver(
    self,
    selector: #selector(handleAppActivation(_:)),
    name: NSWorkspace.didActivateApplicationNotification,
    object: nil
)

@objc private func handleAppActivation(_ note: Notification) {
    guard let app = note.userInfo?[NSWorkspace.applicationUserInfoKey]
        as? NSRunningApplication else { return }
    panel.level = app.processIdentifier == ProcessInfo.processInfo.processIdentifier
        ? .floating : .normal
}

NSWorkspace.didActivateApplicationNotificationは、自分を含むどのアプリがアクティブになっても 発火する。userInfoからNSRunningApplicationを取り出し、 そのプロセスIDが自分自身のもの(ProcessInfo.processInfo.processIdentifier)かどうかで、 「今アクティブなのは自分か、他のアプリか」を判定している。
