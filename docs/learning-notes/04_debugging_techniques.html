デバッグ技法編: GUIアプリのブラックボックスを可視化する

Xcodeのデバッガに頼らず、コマンドラインだけでmacOSのGUIアプリをビルド・実行・検証したときに使った手法をまとめます。

1. Xcode無しでの.appバンドル生成

FloatPlayerはSwift Package Managerだけで作っており、swift buildが生成するのは ただの実行可能バイナリで、正式な.appバンドルではない。これが原因で、IME(日本語入力などの 文字入力を管理する仕組み)がプロセスを正しく認識できず、テキストフィールドの挙動が不安定になる問題があった。 

# swift runの生バイナリで出ていたエラー
error messaging the mach port for IMKCFRunLoopWakeUpReliable

解決策は、ビルド成果物を手動で.appバンドルの形(Contents/MacOS/に実行可能バイナリ、 Contents/Info.plistにメタ情報)に組み立て、アドホック署名するスクリプトを自作すること。 

#!/bin/bash
swift build -c "$CONFIG"
mkdir -p "$APP_NAME/Contents/MacOS"
cp ".build/$CONFIG/$EXECUTABLE_NAME" "$APP_NAME/Contents/MacOS/$EXECUTABLE_NAME"
cp Info.plist "$APP_NAME/Contents/Info.plist"
# アドホック署名しておく(IME/TCCまわりの認識を安定させるため)
codesign --force --deep --sign - "$APP_NAME"

codesign --sign -の-は「アドホック署名(自己署名、配布はできないが実行は可能)」を意味する。 Apple Developer Programに入っていなくても、ローカルで動かすだけならこれで十分。

2. 統合ログ(log show)でOSレベルの原因を特定する

自動終了バグの真因を突き止めたのは、macOSの統合ログシステムを 直接調べたことだった。log showコマンドは、Console.appの検索窓と同じ内容をターミナルから クエリできる。 

/usr/bin/log show --style compact --last 5m \
  --predicate 'processImagePath contains "FloatPlayer"'

--predicateには、プロセス名・サブシステム・メッセージ内容などで絞り込む条件式を書ける。 「自分のアプリに関係するログだけ」に絞ることで、システム全体の膨大なログの中から terminate:やAttempting sudden terminationのような、AppKit内部の動きを 直接観測できた。

Tips: TCC(プライバシー許可)関連のログも同じ方法で追える

スクリーンショット貼り付け機能でクリップボード読み取りが期待通り動かなかったとき、 process == "tccd"で絞り込むと、パーミッション要求(TCCAccessRequest)の やり取りがそのままログに出てくる。 

/usr/bin/log show --last 5m --predicate 'process == "tccd"'

3. tccutilで権限状態をリセットする

一度「許可しない」を選んでしまった権限や、テスト中に不安定になった権限状態をリセットしたいとき、 tccutilコマンドが使える。 

tccutil reset All com.haruboo.floatplayer

これで指定したbundle identifierに紐づくすべての権限許可状態が消え、次回そのAPIを使ったときに 改めて許可ダイアログが出るようになる(Allの部分をScreenCaptureのような サービス名にすれば特定の権限だけをリセットすることも可能)。

注意: アドホック署名は毎回「別アプリ」扱いされることがある

ビルドのたびにcodesign --sign -でアドホック署名をやり直すと、コード自体の ハッシュ値が毎回変わる。TCCは「コード署名の同一性」で権限を紐づけているため、 リビルドするたびにデスクトップアクセスなどの許可ダイアログが再度出てくることがある。 これは正規のDeveloper ID証明書で安定した署名をしていない、ローカルビルド特有の制約。

4. スクリーンショットを使った「自分で見る」検証ループ

GUIアプリの見た目の問題(レイアウト崩れ、ウィンドウサイズなど)は、コードを読むだけでは分からないことが多い。 screencaptureコマンドを使えば、ターミナルから直接、今画面に映っているものをキャプチャできる。 

screencapture -x /tmp/check.png   # -x: シャッター音を鳴らさない

これにより、「コードを直す → ビルドする → アプリを再起動する → スクリーンショットで見た目を確認する」 というループを、ユーザーに毎回確認を頼まずに自分で何周も回すことができた。

ハンズオン: 動画の自動録画

screencaptureは静止画だけでなく、-vオプションで動画も撮れる。 ただし動画モードは既定では手動停止が前提のため、バックグラウンド実行してから SIGINTを送ることで、スクリプトから時間指定の録画ができる。 

screencapture -v /tmp/demo.mov &
REC_PID=$!
sleep 24
kill -INT $REC_PID

紹介動画の素材作りでは、実際にマウス操作を自動化する権限(Accessibility)がなかったため、 「アプリ側に一定時間ごとに状態を切り替える無人デモシーケンスを環境変数で仕込み、それを録画する」 という迂回策を取った。GUIの自動操作ができない制約下でも、操作される側(アプリ)に タイマー駆動のシナリオを実装することで、擬似的な自動デモ録画を実現できる。

5. 「実測」を信じ、目視を疑う

このプロジェクトで最も時間がかかったウィンドウ巨大化バグ(詳細)では、 スクリーンショットの目視だけでは「本当にウィンドウが大きいのか、他のウィンドウと重なって見えているだけか」 を何度も見誤った。最終的に効いたのは、printでpanel.frameの値そのものを ログに出す、という最も原始的な方法だった。

GUIのバグ調査は「目に見える現象」と「実際に起きていること」がズレやすい。可能な限り、 数値・ログという反証可能な形で状態を確認するのが結局は一番早い、という教訓が今回の一番の学びだった。
