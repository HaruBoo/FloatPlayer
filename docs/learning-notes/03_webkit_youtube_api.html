WebKit/YouTube API編: WKWebViewでYouTubeを埋め込む

WKWebViewでYouTubeを再生する際に踏んだ2つの埋め込みエラーと、公式のIFrame Player APIによるJS制御、YouTube Data APIでのチャプター抽出を解説します。

1. なぜ直接ダウンロードせず埋め込みなのか

FloatPlayerはYouTube動画をダウンロードしない。WKWebView(SafariやChromeと同じWebKitエンジン)の中で、 YouTube公式の埋め込みプレイヤーをそのまま動かしている。これはYouTubeの利用規約に沿った標準的な埋め込み方法で、 一般的なWebサイトが動画を埋め込むのとまったく同じ仕組み。

2. バグ1: 「動画プレーヤーの設定エラー」— iframeなしでの直接ナビゲーション

最初の実装では、https://www.youtube.com/embed/{動画ID}というURLに WKWebViewを直接ナビゲートしていた。 

// うまくいかなかった実装
webView.load(URLRequest(url: embedURL))

一見動きそうに見えるが、実際に再生しようとすると「動画プレーヤーの設定エラーが発生しました」という エラーが表示された。原因は、YouTubeの埋め込みプレイヤーが「自分はiframeの中で動いている」という前提で いくつかの検証(親ウィンドウのオリジンチェックなど)を行っており、WKWebViewの最上位ページとして直接開いた場合、 その前提が崩れて検証に失敗するため。

直し方: ラッパーHTMLの中にiframeとして埋め込む

実際のWebサイトがYouTube動画を埋め込むときと同じ状態を再現するため、 自前のHTMLページを作り、その中に<iframe>として埋め込むよう変更した。 

let html = """
<!doctype html>
<html><body style="margin:0;background:transparent;">
<iframe src="\(embedURL.absoluteString)"
        style="position:absolute;top:0;left:0;width:100%;height:100%;border:0;"
        allow="autoplay; encrypted-media"
        allowfullscreen></iframe>
</body></html>
"""
webView.loadHTMLString(html, baseURL: URL(string: "https://example.com"))

loadHTMLString(_:baseURL:)は「文字列で組み立てたHTMLを、指定したURLがベースであるかのように」 読み込ませるAPI。これにより、YouTubeの埋め込みプレイヤーから見ると「あるWebページのiframeの中で 動いている」という、想定通りの状態になる。

3. バグ2: エラーコード152-4 — baseURLの選び方

iframe化しただけではまだ再生できなかった。今度は「エラーコード152-4 この動画は再生できません」 という、別の動画でも再現するエラーが出た。原因はbaseURLにhttps://www.youtube.comを 指定していたこと。

実際のWebサイトがYouTube動画を埋め込むとき、埋め込み元のサイトがyoutube.com自身であることは 普通ありえない。baseURLをYouTube自身のドメインにすると、YouTube側の検証ロジックが 「自己参照的な埋め込み」と判断して不審に扱い、再生を拒否していたと考えられる。 無関係な独自ドメイン風のbaseURLに変更したところ、再生できるようになった。 

// NG: 自己埋め込みとして扱われて再生に失敗する
webView.loadHTMLString(html, baseURL: URL(string: "https://www.youtube.com"))

// OK: 実際のサイトが埋め込む状況に近い、無関係なオリジンにする
webView.loadHTMLString(html, baseURL: URL(string: "https://floatplayer.local"))
Tips: 「動くはずのコードが動かない」時はエラーメッセージの版元を疑う

今回の2つのエラーはどちらも、YouTube埋め込みプレイヤー自身のJavaScriptが出しているエラーであり、 Swift/WKWebView側には何のエラーも出ない(読み込み自体は「成功」する)。GUI上でエラーが出ても、 それが自分のアプリのバグなのか、埋め込んだWebコンテンツ側の検証ロジックなのかを見極めることが重要。

4. YouTube IFrame Player APIでJS制御する

当初は「iframeのURLにstart=秒数を付けて再読み込みする」ことでチャプタージャンプを実装していたが、 再読み込みのたびに映像が一瞬止まり、モード切り替えのたびにWKWebView自体が作り直されて再生位置が失われる という問題があった。これを解決したのが、YouTube公式のIFrame Player API。 

<div id="player"></div>
<script>
  var tag = document.createElement('script');
  tag.src = "https://www.youtube.com/iframe_api";
  document.body.appendChild(tag);

  var player;
  function onYouTubeIframeAPIReady() {
    player = new YT.Player('player', {
      videoId: '\(videoID)',
      playerVars: { autoplay: 1, playsinline: 1, loop: 1, playlist: '\(videoID)' }
    });
  }
  function fpPause() { if (player) player.pauseVideo(); }
  function fpPlay()  { if (player) player.playVideo(); }
  function fpSeek(sec) { if (player) { player.seekTo(sec, true); player.playVideo(); } }
</script>

https://www.youtube.com/iframe_apiを読み込むと、ページの中に YT.PlayerというJavaScriptオブジェクトが使えるようになる。これは単なる<iframe>タグを 自前で書くよりも一段抽象度が高く、pauseVideo() / playVideo() / seekTo()といった メソッドでページの再読み込みなしに再生を制御できる。

Swift側からはWKWebView.evaluateJavaScript(_:)で、このfpPause()のような 自作のラッパー関数を呼び出す。 

webView.evaluateJavaScript("fpSeek(\(seconds));", completionHandler: nil)
ハンズオン: モード切り替えでも再生を止めない設計

SwiftUIで「YouTube/写真/動画」をタブのように切り替える場合、素朴にswitch文で 表示するビューを出し分けると、非表示になったタブのビュー(WKWebViewやAVPlayer)は SwiftUIのツリーから外れて破棄されてしまう。これだと裏に回ったYouTubeの音が鳴り続けたり、 戻ってきたときに動画が最初から再生されたりする不具合につながる。

FloatPlayerでは3つのセクションをZStackで常に重ねてマウントしたままにし、 .opacity()と.allowsHitTesting()だけで見た目上の表示/非表示を切り替えている。 

ZStack {
    youtubeSection
        .opacity(viewModel.mode == .youtube ? 1 : 0)
        .allowsHitTesting(viewModel.mode == .youtube)
    photoSection
        .opacity(viewModel.mode == .photo ? 1 : 0)
        .allowsHitTesting(viewModel.mode == .photo)
    videoSection
        .opacity(viewModel.mode == .video ? 1 : 0)
        .allowsHitTesting(viewModel.mode == .video)
}

これによりWKWebViewのインスタンスは維持されたまま裏に回るだけなので、 モードが変わったタイミングをviewModel.$modeで購読してfpPause() / fpPlay()を呼べば、「他のモードに切り替えたら一時停止、戻したら続きから再開」という 自然な挙動が実現できる。

5. YouTube Data APIでチャプターを抽出する

動画の概要欄には、投稿者が書いた「0:00 イントロ」のようなタイムスタンプ行がしばしば含まれる。 これをチャプターとして使うため、YouTube Data API v3のvideosエンドポイントで 概要欄のテキストを取得し、正規表現で解析している。 

nonisolated static func parseChapters(fromDescription description: String) -> [Chapter] {
    let pattern = #"^\(?(\d{1,2}(?::\d{2}){1,2})\)?\s*[-:–—]?\s*(.+)$"#
    guard let regex = try? NSRegularExpression(pattern: pattern) else { return [] }

    var results: [Chapter] = []
    for rawLine in description.components(separatedBy: .newlines) {
        // 行頭が "0:00" や "1:23:45" のような時刻表記の行だけ拾う
        // ...
    }
    // 誤検出(本文中の時刻表記など)を避けるため、最低2件そろって初めてチャプターとみなす
    return results.count >= 2 ? results : []
}

正規表現(\d{1,2}(?::\d{2}){1,2})は「1〜2桁の数字の後に、:2桁が1〜2回続く」という意味で、 0:00(分:秒)にも1:23:45(時:分:秒)にも両方マッチする。 本文中にたまたま出てくる数字列を誤ってチャプターと判定しないよう、「行の先頭」に固定し(^)、 かつ「2件以上そろわなければチャプター扱いしない」という安全マージンを設けている。

注意: nonisolated staticにする理由

parseChaptersはURLSessionのコールバック(バックグラウンドスレッド)から呼ばれる。 しかしMediaViewModelクラス自体は@MainActorなので、そのままでは 「バックグラウンドからメインアクター専用のメソッドを呼んでいる」という警告になる。 このメソッドはselfの状態を一切使わない純粋関数なので、明示的に nonisolatedを付けてアクター制約から外している。結果をMainActorへ戻すのは 呼び出し側の責務にする、という役割分担。 

URLSession.shared.dataTask(with: url) { [weak self] data, _, _ in
    let parsed = Self.parseChapters(fromAPIResponse: data)  // バックグラウンドで実行
    Task { @MainActor in
        self?.chapters = parsed  // メインアクターに戻してから状態を更新
    }
}.resume()
