Swift基礎編: 言語機能とAppKitの土台

FloatPlayerで実際に使ったSwiftの言語機能と、SwiftUI/AppKitを橋渡しする仕組みを、コードと対応させながら解説します。

0. Swiftの基礎知識(ざっくり)

本編に入る前に、このアプリのコードを読むうえで最低限知っておくと楽になるSwiftの基礎を、 ごく軽くおさらいします。すでに知っている場合は1章まで読み飛ばして構いません。

let と var

letは再代入できない定数、varは再代入できる変数を宣言します。 Swiftでは「変わらない値にはletを使う」のが強く推奨されるスタイルで、 このプロジェクトでもvarは「本当に変化する状態」(例:@Published var mode)にだけ使っています。 

let apiURL = "https://www.googleapis.com/youtube/v3/videos" // 再代入不可
var isLoading = false                                       // 再代入可能
isLoading = true

型推論とOptional(nilを許す型)

Swiftはlet name = "FloatPlayer"のように書くだけで型(この場合String)を推論してくれます。 型注釈を省略できますが、型自体は静的に決まっている(後から別の型を代入することはできない)強い型付け言語です。

Swiftの大きな特徴がOptional。「値があるかもしれないし、無い(nil)かもしれない」ことを 型のレベルで表現する仕組みで、型名の後ろに?を付けます。 

var videoURL: URL?          // URLが入っているか、nilかのどちらか
var currentVideoID: String? // 動画未選択の状態をnilで表現している

Optionalの中身を安全に取り出すにはif letやguard letを使います。 

guard let id = Self.extractYouTubeID(from: youtubeInput) else { return }
// ここから先では id は String?ではなく、中身のあるString として使える

guard ... else { return }は「条件を満たさなければここで処理を打ち切る」という書き方で、 ネストを深くせずに「早期リターン」できるためSwiftでは非常によく使われます。

クロージャ(無名関数)

クロージャは{ 引数 in 処理 }という形の「その場で作る関数」です。他の言語のラムダ式・ アロー関数に相当します。非同期処理のコールバックとして頻繁に登場します。 

URLSession.shared.dataTask(with: url) { data, _, _ in
    // このブロック自体がクロージャ。dataが返ってきたときに後から呼ばれる
    print(data ?? "no data")
}.resume()

クロージャが後から実行される(今すぐではない)関数の引数になっているとき、Swiftはその引数に @escapingという注釈を要求することがあります。「この関数を抜けた後もクロージャが生き残る」 ことを明示するための注釈です。

struct と class の違い

Swiftには値型のstructと参照型のclassの両方があります。 structは代入・関数への引渡し時にコピーされますが、classは 同じ実体への参照が共有されます。SwiftUIのビュー(ContentView)や Chapterのようなデータの入れ物はstruct、状態を持ち続けて UI全体で共有したいMediaViewModelはclass、という使い分けをしています。 

struct Chapter: Identifiable {   // 値型: コピーされる、単純なデータ
    let id = UUID()
    let seconds: Int
    let title: String
}

final class MediaViewModel: ObservableObject {  // 参照型: 複数箇所で同じインスタンスを共有
    @Published var mode: PlaybackMode = .youtube
}

enum(列挙型)

再生モードのように「決まった選択肢のどれか」を表すのにenumを使います。 Swiftのenumは他言語と違い、計算プロパティやメソッドを持てるのが特徴です。 

enum PlaybackMode: String, CaseIterable, Identifiable {
    case youtube, photo, video

    var label: String {
        switch self {
        case .youtube: return "YouTube"
        case .photo: return "写真"
        case .video: return "動画"
        }
    }
}

プロトコル(protocol)

プロトコルは「このメソッド/プロパティを持つこと」という約束事を定義するもので、 他言語のインターフェースに近い概念です。AppDelegateがNSApplicationDelegateプロトコルに 準拠する(: NSObject, NSApplicationDelegate)ことで、AppKit側から applicationDidFinishLaunchingなどの決まったタイミングで呼び出してもらえるようになります。 

final class AppDelegate: NSObject, NSApplicationDelegate {
    func applicationDidFinishLaunching(_ notification: Notification) {
        // アプリ起動完了時にAppKitから呼ばれる
    }
}
Tips: ここまでの基礎が分かれば読み進められる

let/var、Optional、クロージャ、struct/class、enum、protocolの6つが分かれば、 このアプリのコードの9割は読めるようになります。ここから先は、それらの基礎知識を前提に、 このアプリならではの実践的な使い方を見ていきます。

1. @MainActor とアクター分離

Swift 6ではデータ競合をコンパイル時に検出する「厳格な並行性チェック」が既定で有効になっている。 AppKit/SwiftUIのAPIはほぼすべて「メインスレッドでしか呼んではいけない」という制約があり、 Swiftはこれを@MainActorという属性で型システムに組み込んでいる。 

@MainActor
final class AppDelegate: NSObject, NSApplicationDelegate {
    private var panel: NSPanel!
    // ...
}

クラス全体に@MainActorを付けると、そのクラスのすべてのプロパティ・メソッドが 「メインスレッド上でのみアクセス可能」という制約を持つようになる。これにより、 バックグラウンドスレッドから誤ってpanelを触ってしまうようなコードはコンパイルエラーになる。

実際に踏んだエラー: main.swiftトップレベルの分離違反

main.swiftのトップレベル(スクリプトのように直接実行される部分)は、 実行時にはメインスレッド上で動くにもかかわらず、型システム上は@MainActorに分離されていない。 そのため、@MainActor指定したAppDelegate()を素朴に呼ぶと次のエラーになった。 

error: call to main actor-isolated initializer 'init()'
in a synchronous nonisolated context

解決策はMainActor.assumeIsolatedで「ここは実際にはメインスレッドである」と明示的に橋渡しすること。

import AppKit

let app = NSApplication.shared
// main.swiftのトップレベルはmain actor分離されていないため、明示的に橋渡しする
let delegate = MainActor.assumeIsolated { AppDelegate() }
app.delegate = delegate
app.setActivationPolicy(.regular)
app.run()

2. Combineによる状態管理(@Published)

FloatPlayerではSwiftUIの画面とAppKitのウィンドウ・メニューバーという「2つのUI」が 同じ状態を参照する必要がある(例: 透明度スライダーの値は、SwiftUIのSliderにもAppKitのNSMenuItemにも影響する)。 この橋渡しにCombineの@Publishedとsinkを使っている。 

@MainActor
final class MediaViewModel: ObservableObject {
    @Published var mode: PlaybackMode = .youtube
    @Published var isUIHidden: Bool = false
    @Published var mediaOpacity: Double = 1.0
    // ...
}

ObservableObjectに準拠したクラスの@Publishedプロパティが変化すると、 SwiftUI側は自動的に再描画される。それとは別に、AppKit側(AppDelegate)では sinkで変化を購読し、ステータスバーのメニュー項目などAppKitのAPIを手動で更新する。 

viewModel.$isUIHidden
    .receive(on: RunLoop.main)
    .sink { [weak self] hidden in
        self?.uiHiddenMenuItem.state = hidden ? .on : .off
        self?.uiHiddenMenuItem.title = hidden ? "UIを表示" : "UIを隠す"
    }
    .store(in: &cancellables)

[weak self]は「このクロージャがselfを弱参照で持つ」という指定。sinkのクロージャは cancellables(Set<AnyCancellable>)に保持され続けるので、もし強参照(selfをそのまま書く)にすると selfがこのクロージャを持ち、クロージャがselfを持つという循環参照(メモリリーク)になる。weakで防いでいる。

Tips: PassthroughSubjectで「命令」を流す

チャプター選択時のシーク処理には、状態を表す@Publishedではなく PassthroughSubject<Int, Never>を使っている。 

let seekSubject = PassthroughSubject()

func seek(toChapter chapter: Chapter) {
    seekSubject.send(chapter.seconds)
}

@Publishedは「現在の状態」を表すのに向いているが、シークは「一回きりのイベント」であって 状態ではない(同じ秒数に2回シークしたいこともある)。イベント通知には値を保持しないPassthroughSubjectの方が適している。

3. SwiftUIとAppKitの橋渡し

NSHostingViewでSwiftUIをAppKitウィンドウに載せる

FloatPlayerはWindowGroupのような通常のSwiftUI Appライフサイクルを使わず、 NSPanelを直接作って、そのcontentViewにSwiftUIビューを載せている。 この橋渡し役がNSHostingView。 

let hosting = NSHostingView(rootView: ContentView(viewModel: viewModel))
newPanel.contentView = hosting

これにより、「タイトルバーを消す」「常に最前面に浮かせる」「クリックスルーさせる」といった AppKit特有の細かいウィンドウ制御をしながら、画面の中身はSwiftUIで書けるようになる。 (このNSHostingViewには重大な落とし穴があり、詳しくはNSWindow/NSPanel編で扱う。)

NSViewRepresentableでAppKitの部品をSwiftUIに載せる

逆方向の橋渡しがNSViewRepresentable。YouTube再生にはWKWebView(AppKit/WebKitのクラス)が必要だが、 SwiftUIには標準のWebView部品がないため、自分でラップする。 

struct YouTubeWebView: NSViewRepresentable {
    @ObservedObject var viewModel: MediaViewModel

    func makeNSView(context: Context) -> WKWebView {
        let webView = WKWebView(frame: .zero, configuration: config)
        // ...
        return webView
    }

    func updateNSView(_ webView: WKWebView, context: Context) {
        // SwiftUIの状態が変わるたびに呼ばれる
    }

    static func dismantleNSView(_ webView: WKWebView, coordinator: Coordinator) {
        // このビューがSwiftUIツリーから外れるときに呼ばれる
        webView.pauseAllMediaPlayback(completionHandler: nil)
    }
}

makeNSView(初回生成)、updateNSView(SwiftUIの再描画のたびに呼ばれる)、 dismantleNSView(ビューが破棄されるとき)という3つのライフサイクルメソッドを実装するのが基本形。

ハンズオン: Coordinatorパターンで外部からの命令を受け取る

「チャプターを選んだら、再読み込みせずに今再生中のWKWebViewをシークする」ためには、 SwiftUIの外(メニューバーのクリック)からWKWebViewに命令を送る経路が必要になる。 NSViewRepresentableはCoordinatorという補助オブジェクトを持てるので、 そこでCombineの購読を持たせている。 

final class Coordinator {
    private weak var webView: WKWebView?
    private var cancellables = Set()

    func attach(_ webView: WKWebView) {
        self.webView = webView
        viewModel.seekSubject
            .sink { [weak self] seconds in
                self?.run("fpSeek(\(seconds));")
            }
            .store(in: &cancellables)
    }
}

こうしておくと、AppDelegate側は「seekSubjectに値を流すだけ」で、実際にWKWebViewの JavaScriptを実行する責務はCoordinatorに閉じ込められる。SwiftUIの外の世界(メニューバー)と 中の世界(WKWebView)を、CombineのSubjectを介して疎結合につなぐパターン。

4. DispatchSourceによるファイル監視

スクリーンショットの自動検出には、ポーリングではなくカーネルのイベント通知(kqueue)を 使った軽量な仕組みを使っている。 

directoryFD = open(dir.path, O_EVTONLY)
let source = DispatchSource.makeFileSystemObjectSource(
    fileDescriptor: directoryFD,
    eventMask: .write,
    queue: .main
)
source.setEventHandler { [weak self] in
    self?.checkForNewFiles(in: dir)
}
source.resume()

open(path, O_EVTONLY)は「読み書きはしないが、変化だけ監視したい」ときのフラグ付きオープン。 ディレクトリのファイルディスクリプタに対して.writeイベントを監視すると、 そのディレクトリの中身が変化した瞬間(ファイルの追加・削除・リネーム)にハンドラが呼ばれる。 これはPythonのwatchdogやNode.jsのfs.watchと同じ、OSネイティブのファイル監視API。

注意: 差分検出はアプリ側で行う必要がある

.writeイベントは「何かが変わった」としか教えてくれない。「何が追加されたか」は 自分でディレクトリの中身を列挙して、前回保持していたファイル名の集合(Set<String>)と 差分(subtracting)を取る必要がある。 

let current = Self.listImageFiles(in: dir)
let newNames = current.subtracting(knownFiles)
knownFiles = current
実際に踏んだエラー: イベント発火直後は書き込みが終わっていないことがある

.writeイベントは「書き込みが完了した後」ではなく「書き込みが始まった/進行した」タイミングでも 発火することがある。スクリーンショットのように大きな画像ファイルだと、イベントを受けた直後に NSImage(contentsOf:)で読み込もうとすると、まだファイルが書き終わっておらずデコードに失敗することがあった。 

// 書き込みが完了しきっていないタイミングで読むと失敗するため少し待つ
DispatchQueue.main.asyncAfter(deadline: .now() + 0.3) { [weak self] in
    self?.onNewScreenshot(newest)
}

根本的な解決(ファイルサイズが安定するまで待つ、など)もできるが、今回は「0.3秒待ってから読む」という 簡易的なデバウンスで十分実用的な精度が出た。

5. 機能: 透明度スライダーを「映像用」と「UI用」に分ける

FloatPlayerには透明度スライダーが2つある。mediaOpacityはYouTube/写真/動画そのものの透明度、 uiOpacityは上下バーやスライダーなどUI部品の透明度で、それぞれ独立に0.15〜1.0の範囲で動かせる。 

// MediaViewModel.swift
@Published var mediaOpacity: Double = 1.0 // YouTube/写真/動画の透明度
@Published var uiOpacity: Double = 1.0    // 上下バーなどUIの透明度

実装としては、SwiftUIの.opacity()修飾子を「どの階層のViewに付けるか」を厳密に分けるだけ。 上下バー全体にはuiOpacityを、実際に映る中身(YouTubeWebView・Image・ VideoPlayer)にだけmediaOpacityを付けている。 

// ContentView.swift
topBar
    .background(DragHandle())
    .opacity(viewModel.uiOpacity)
// ...
YouTubeWebView(viewModel: viewModel)
    .opacity(viewModel.mediaOpacity)
実際に踏んだバグ: 説明文や入力欄まで透明になってしまう

最初の実装では、YouTubeの入力欄やヒントテキストを含むセクション全体に mediaOpacityを1つだけ付けていた。これだと映像を透明にしようとスライダーを動かすたびに、 TextField(URL入力欄)や説明文まで一緒に薄くなり、「文字が読めない」「入力欄がどこにあるか 分からない」状態になってしまった。 

// NG: セクション全体に1つのopacityを付けていた
youtubeSection
    .opacity(viewModel.mediaOpacity)

原因はシンプルで、「透明にしたいのは映像だけ」という意図に対して、.opacity()を 付ける階層が広すぎたこと。修正は、入力欄やヒントテキストを囲む親Viewから.opacity()を外し、 実際の映像コンポーネント(YouTubeWebView・Image・VideoPlayer)にだけ 個別に付け直すことだった。 

// OK: 入力欄・説明文には付けず、映像そのものにだけ付ける
if !viewModel.isUIHidden {
    TextField("YouTubeのURLまたは動画ID", text: $viewModel.youtubeInput)
    // opacityなし。常にはっきり読める
}
if viewModel.currentVideoID != nil {
    YouTubeWebView(viewModel: viewModel)
        .opacity(viewModel.mediaOpacity) // 透明にしたいのはここだけ
}

SwiftUIの.opacity()は子View全体に効くため、「何を透明にしたいか」を Viewツリーのどの階層に付けるかで正確にコントロールする必要がある、という教訓を得た。
