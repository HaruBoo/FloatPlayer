デバッグ技法編: GUIアプリのブラックボックスを可視化する

Xcodeのデバッガに頼らず、コマンドラインだけでmacOSのGUIアプリをビルド・実行・検証したときに使った手法をまとめます。

1. Xcode無しでの.appバンドル生成

FloatPlayerはSwift Package Manager(Swift基礎編0章 で触れた、Xcodeなしでコマンドラインだけでビルドできる仕組み)だけで作っています。 swift buildコマンドを実行すると、ソースコードがコンパイル(人間が読める形のコードを、 コンピュータが実行できる機械語に変換すること)されて実行可能バイナリ(ダブルクリックや コマンドで直接実行できる、機械語のファイル)が生成されますが、これは正式な.appバンドルではありません。

.appバンドルとは、macOSアプリの実体で、実は中身は「フォルダ」です(Finderでは1つのアイコンに 見えますが、右クリックの「パッケージの内容を表示」で開くと、中に複数のファイルが入っている フォルダだと分かります)。実行可能バイナリ本体だけでなく、アプリの名前・バージョン・アイコンなどの メタ情報を書いたInfo.plistというファイルも一緒に、決まった構造で入っている必要があります。 swift buildはこの構造を作ってくれないため、ただのバイナリを直接実行すると、 macOSの一部の仕組みが「これは正式なアプリではない」と認識してしまいます。実際、IME(日本語入力などの 文字入力を管理する、macOS標準の仕組み)がプロセス(実行中のプログラムの単位)を正しく認識できず、 テキストフィールドの挙動が不安定になる問題が起きました。 

# swift runの生バイナリで出ていたエラー
error messaging the mach port for IMKCFRunLoopWakeUpReliable

解決策は、ビルド成果物を手動で.appバンドルの形(Contents/MacOS/フォルダに 実行可能バイナリを、Contents/Info.plistにメタ情報を配置する、Appleが定めた決まった構造)に 組み立て、コード署名(このアプリが誰によって作られ、改ざんされていないことをOSに証明するための 電子的な印)を付けるスクリプトを自作することでした。 

#!/bin/bash
swift build -c "$CONFIG"
mkdir -p "$APP_NAME/Contents/MacOS"
cp ".build/$CONFIG/$EXECUTABLE_NAME" "$APP_NAME/Contents/MacOS/$EXECUTABLE_NAME"
cp Info.plist "$APP_NAME/Contents/Info.plist"
# アドホック署名しておく(IME/TCCまわりの認識を安定させるため)
codesign --force --deep --sign - "$APP_NAME"

codesignはmacOS標準のコード署名コマンドです。--sign -の-は 「アドホック署名」(正規の証明書を使わない自己署名で、他人への配布はできないが、自分のMacで 実行するだけなら十分)を意味します。Apple Developer Program(年会費を払って正規の証明書を取得する 仕組み)に入っていなくても、ローカルで動かすだけならこれで十分です。

2. 統合ログ(log show)でOSレベルの原因を特定する

自動終了バグの真因を突き止めたのは、macOSの統合ログ システム(OS自身やあらゆるアプリが出力するログを、1つの仕組みにまとめて記録・検索できるようにした もの)を直接調べたことでした。自分のアプリにprintを書き足すだけでは、OS内部(AppKitの 中身など、自分では書いていない部分)で何が起きているかまでは分かりません。そこで、OS自体が既に 記録しているログを覗きにいく、というアプローチを取りました。log showコマンドは、 Console.app(macOS標準のログ閲覧アプリ)の検索窓に打ち込むのと同じ内容を、ターミナルからコマンドで クエリ(検索条件を指定してデータを問い合わせること)できます。 

/usr/bin/log show --style compact --last 5m \
  --predicate 'processImagePath contains "FloatPlayer"'

--predicate(述語、つまり「条件式」)には、プロセス名・サブシステム(ログを出している 機能のまとまり)・メッセージ内容などで絞り込む条件を書けます。--last 5mは「直近5分間」 という意味です。「自分のアプリに関係するログだけ」に絞ることで、システム全体の膨大なログの中から、 terminate:(終了処理)やAttempting sudden termination(即時終了を試みている、 という意味のAppKit内部のメッセージ)のような、AppKit内部の動きを直接観測できました。

この「OSが元々記録しているログを、条件を絞って読みにいく」という考え方は、自分のコードに デバッグ用の出力を仕込む余地がない場面(OS内部・他人が書いたライブラリ内部など)で原因を探るときに、 広く応用できます。

Tips: TCC(プライバシー許可)関連のログも同じ方法で追える

TCC(Transparency, Consent, and Control の略)は、macOSがカメラ・マイク・クリップボードの 読み取りなど、プライバシーに関わる機能へのアクセスをアプリごとに許可/拒否管理している仕組みです。 スクリーンショット貼り付け機能でクリップボード読み取りが期待通り動かなかったとき、 process == "tccd"(TCCを管理しているOSの裏方プロセスの名前)で絞り込むと、 パーミッション要求(TCCAccessRequest)のやり取りがそのままログに出てきました。 

/usr/bin/log show --last 5m --predicate 'process == "tccd"'

3. tccutilで権限状態をリセットする

一度「許可しない」を選んでしまった権限や、テスト中に不安定になった権限状態をリセットしたいとき、 tccutilコマンド(前章のTCCを、コマンドラインから直接操作するためのツール)が使えます。 

tccutil reset All com.haruboo.floatplayer

com.haruboo.floatplayerの部分はbundle identifier(バンドルID、アプリ1つ1つに 付ける「逆ドメイン形式」の一意な名前。ドメインを逆順にしてcom.会社名.アプリ名のように 書く慣習がある)です。このコマンドを実行すると、指定したbundle identifierに紐づくすべての 権限許可状態が消え、次回そのAPIを使ったときに改めて許可ダイアログが出るようになります (Allの部分をScreenCaptureのようなサービス名にすれば、特定の権限だけを リセットすることも可能です)。

注意: アドホック署名は毎回「別アプリ」扱いされることがある

ビルドのたびにcodesign --sign -でアドホック署名をやり直すと、コード自体の ハッシュ値(ファイルの中身から計算される、内容が1バイトでも変わると別の値になる指紋のような 数値)が毎回変わります。TCCは「コード署名の同一性」、つまりこのハッシュ値をもとにアプリを識別して 権限を紐づけているため、リビルドするたびに、OSからは「別のアプリになった」と見なされ、 デスクトップアクセスなどの許可ダイアログが再度出てくることがあります。これは正規のDeveloper ID 証明書で安定した署名をしていない、ローカルビルド特有の制約です。

4. スクリーンショットを使った「自分で見る」検証ループ

GUIアプリの見た目の問題(レイアウト崩れ、ウィンドウサイズなど)は、コードを読むだけでは分からないことが 多いです。screencaptureコマンド(macOS標準の、画面をキャプチャするコマンドラインツール。 Cmd+Shift+4のショートカットが裏で呼んでいるのと同じ機能)を使えば、ターミナルから直接、今画面に 映っているものをキャプチャできます。 

screencapture -x /tmp/check.png   # -x: シャッター音を鳴らさない

これにより、「コードを直す → ビルドする → アプリを再起動する → スクリーンショットで見た目を確認する」 というループを、人に毎回確認を頼まずに自分で何周も回すことができました。この「変更 → 実行 → 結果を確認、を素早く繰り返す」という考え方自体は、自動テストを書く場合にも共通する基本姿勢です。 見た目のように自動テストで判定しにくい部分を検証するときは、こうして目に見える形で証拠を残しながら 確認するのが有効です。

ハンズオン: 動画の自動録画

screencaptureは静止画だけでなく、-vオプションで動画も撮れます。 ただし動画モードは既定では「もう一度実行して手動で停止する」という操作が前提のため、 バックグラウンド実行(処理をターミナルの入力を占有しない裏側で走らせること。コマンド末尾の &で指定する)してから、一定時間後にシグナル(実行中のプロセスに送る、 「終了して」のような短い合図。OSの標準的な仕組み)の1つであるSIGINT(Ctrl+Cを押したときと 同じ「中断」の合図)を送ることで、スクリプトから時間を指定した録画ができます。 

screencapture -v /tmp/demo.mov &
REC_PID=$!
sleep 24
kill -INT $REC_PID

$!は「直前にバックグラウンドで起動したプロセスのプロセスID(実行中のプログラムに 割り振られる番号)」を表すシェルの特殊変数です。これをREC_PIDという変数に控えておき、 後からkill -INT(そのプロセスIDにSIGINTを送るコマンド)で狙い撃ちして止めています。

紹介動画の素材作りでは、実際にマウス操作を自動化する権限(Accessibility、他のアプリを プログラムから操作するための権限)がなかったため、「アプリ側に一定時間ごとに状態を切り替える 無人デモシーケンスを環境変数で仕込み、それを録画する」という迂回策を取りました。GUIの自動操作が できない制約下でも、操作される側(アプリ自身)にタイマー駆動のシナリオを実装することで、 擬似的な自動デモ録画を実現できます。「外側から操作できないなら、内側にあらかじめ動きを仕込んで おく」という発想の転換は、自動操作の権限やツールが使えない環境で検証や実演をしたいときに 広く応用できます。

5. 「実測」を信じ、目視を疑う

このプロジェクトで最も時間がかかったウィンドウ巨大化バグ(詳細)では、 スクリーンショットの目視だけでは「本当にウィンドウが大きいのか、他のウィンドウと重なって見えているだけか」 を何度も見誤りました。最終的に効いたのは、printでpanel.frame(ウィンドウの 位置とサイズを表すプロパティ)の値そのものをログに出す、という最も原始的な方法でした。

GUIのバグ調査は「目に見える現象」と「実際に起きていること」がズレやすいものです。画面のスクリーン ショットは、あくまで「その瞬間、目にどう映ったか」でしかなく、ウィンドウの実際の座標やサイズという 「事実」とは別物です。可能な限り、数値・ログという反証可能な形(誰が見ても同じ値になり、 「思い込み」の余地がない形、という意味)で状態を確認するのが結局は一番早い、というのが今回の 一番の学びでした。

この考え方は、GUI以外のバグ調査にも共通して応用できます。「なんとなくこう動いているはず」という 推測で止めず、変数の値・ログ・返り値といった具体的な数値や文字列を1つ出力して確認する。 遠回りに見えても、これが結果的には一番の近道になることが多いです。
