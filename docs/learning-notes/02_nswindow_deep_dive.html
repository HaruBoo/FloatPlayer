NSWindow/NSPanel編: 消えるウィンドウと巨大化するウィンドウ

「常に最前面に浮くウィンドウ」と「スクリーンショット専用のフローティングウィンドウ」という2つのNSPanel実装を題材に、その設定内容と、実際に踏んだAppKitのウィンドウ管理まわりの重いバグ5つを解説します。

1. NSPanelの基本設定

FloatPlayerのメインウィンドウは、通常のNSWindowではなくNSPanelで作っています。 NSPanelはNSWindowのサブクラスです。サブクラスとは、既存のクラス (この場合NSWindow、「親クラス」と呼びます)が持つ機能をすべて引き継いだ上で、 独自の機能を追加したクラスのことです。「ウィンドウの基本機能はそのまま使いつつ、 パレット・ツールウィンドウ向けの追加の挙動(後述のnonactivatingPanelなど)も欲しい」 という場合に、サブクラスとして定義されているNSPanelを使います。 

let newPanel = NSPanel(
    contentRect: NSRect(x: 200, y: 200, width: 420, height: 280),
    styleMask: [.titled, .resizable, .closable, .miniaturizable,
                .nonactivatingPanel, .fullSizeContentView],
    backing: .buffered,
    defer: false
)
newPanel.titleVisibility = .hidden
newPanel.titlebarAppearsTransparent = true
newPanel.level = .floating
newPanel.isOpaque = false
newPanel.backgroundColor = .clear

styleMaskの[.titled, .resizable, ...]という書き方は、複数のオン/オフの 設定項目を配列のように並べて一度に指定するためのものです。ここに書かれた項目だけが 有効になり、書かれていない項目は無効(false)として扱われます。それぞれの項目の意味は 次の通りです。

設定意味
.nonactivatingPanelクリックしても他のアプリのアクティブ状態を奪わない
.fullSizeContentViewコンテンツ領域がタイトルバーの裏側まで広がる(独自デザインのタイトルバー風UIを作るときの定番)
level = .floating通常ウィンドウより手前のレイヤーに常に表示する(6章で詳しく扱う)
isOpaque = false / backgroundColor = .clearウィンドウの背景を透明にし、SwiftUI側で自由に描画する
注意: .closable/.miniaturizableを付けないと信号機ボタンが「見えるだけ」になる

.titledを指定すると赤・黄・緑の信号機ボタンは常に描画されます。しかし .closable・.miniaturizableをstyleMaskに含めていないと、 ボタンは表示されるが無効化(グレーアウト)された状態になります。「緑(拡大)だけ効くのに赤と黄が反応しない」 という症状が出たら、まずこのstyleMaskの組み合わせを疑うとよいです。

2. バグ1: 入力しようとするとウィンドウが消える(hidesOnDeactivate)

NSPanelにはhidesOnDeactivateというプロパティ(クラスが持つ設定値・状態のこと)が あり、既定値(初期設定のまま何も変更しなければそうなる値)がtrueになっています。 これは「自分のアプリが非アクティブ(操作対象ではなくなった状態。他のアプリをクリックすると、 今まで触っていたアプリは非アクティブになります)になったら、このパネルを自動的に隠す」という設定です。

FloatPlayerはDock(画面下部などに表示される、起動中アプリのアイコン一覧)に出ない アクセサリ型アプリ(メニューバーの右側に小さいアイコンだけ表示され、Dockには常駐しない タイプのアプリ)として作っていたため、他のアプリ(ターミナルなど)に一瞬フォーカスが移っただけで、 このパネルが勝手に非表示になってしまっていました。ユーザーからは「テキストフィールドをクリックして 入力しようとすると、ウィンドウごと消える」ように見えます。これは、テキストフィールドをクリックした つもりが、その瞬間に一時的に他のアプリ(たとえばIME=日本語入力の変換候補ウィンドウなど)へ フォーカスが移り、それをきっかけにFloatPlayerが「非アクティブになった」と判定されてしまう、 というのが実際の流れでした。 

// NSPanelは既定でhidesOnDeactivate=trueのため、他アプリにフォーカスが移ると
// パネルごと隠れてしまう。常時フロートさせたいのでfalseにする
newPanel.hidesOnDeactivate = false

3. バグ2: システムがアプリを勝手に終了させる(Automatic/Sudden Termination)

hidesOnDeactivateを直しても、まだ「操作しようとすると消える」現象が残っていました。ここからは 推測で直そうとせず、実際に何が起きているかを観察するという方針に切り替えます。 macOSには、アプリが出力したログをまとめて記録している「統合ログ」という仕組みがあり、 log showというコマンド(詳しくはデバッグ技法編)で 過去のログを検索できます。これで実際に何が起きているかを確認したところ、次のログが見つかりました。 

[com.apple.AppKit:Application] terminate:
[com.apple.AppKit:AutomaticTermination] Terminating
[com.apple.AppKit:Application] Attempting sudden termination (1st attempt)

「アプリが勝手に終了(terminate)させられている」ことがログから読み取れます。調べたところ、 macOSには「しばらく使われていないアプリを、ユーザーの許可を得ずに自動終了する」仕組みが Lion(macOS 10.7)以降存在することが分かりました(Automatic Termination / Sudden Terminationという名前の機能です)。これは、メモリや電力を節約するために、 「今使われていなさそうなアプリ」をOSが判断して自動的に終了させる仕組みです。ウィンドウが少ない・ アイドル(何も操作されていない)状態のアプリはこの対象になりやすく、常駐ユーティリティである FloatPlayerは、まさにこの挙動に巻き込まれていました。 

func applicationDidFinishLaunching(_ notification: Notification) {
    // macOSの自動終了/サドンターミネーション機構がこのアプリを
    // アイドルな裏方アプリと誤認して勝手にterminate:するのを防ぐ
    ProcessInfo.processInfo.disableAutomaticTermination("常駐して浮遊表示を続ける必要があるため")
    ProcessInfo.processInfo.disableSuddenTermination()
    // ...
}
実際に踏んだエラー: 自分のデバッグコードが引き金になった別のクラッシュ

この調査中、「何が終了の引き金になったか」を特定するためにapplicationShouldTerminate内で NSApp.currentEventの内容をログ出力するデバッグコードを一時的に追加しました。 

// 危険: currentEventがキーボードイベントとは限らない
print(event.charactersIgnoringModifiers ?? "nil")

ところがこのイベントはキーボードイベントとは限らず(実際はKitDefinedという種類のシステムイベントでした)、 .charactersIgnoringModifiersをキーボード以外のイベント種別に対して呼び出すとAppKit内部で NSInternalInconsistencyExceptionが発生して落ちます。デバッグのために足したログ自体が新たなクラッシュ原因になる という、実務でもよくある教訓を得ました。イベントのtypeを確認してから安全にプロパティへアクセスするよう修正しました。 

if event.type == .keyDown || event.type == .keyUp {
    chars = event.charactersIgnoringModifiers ?? "nil"
} else {
    chars = "n/a"
}

4. バグ3: 最後のウィンドウを閉じるとアプリごと終了する

最終的に、上記の調査ログから本当の引き金が判明しました。デリゲートメソッドとは、 「特定のタイミングでAppKit側から自動的に呼び出される、あらかじめ名前が決まっている メソッド」のことです(Swift基礎編0章の プロトコルの説明も参照)。今回関係していたのは applicationShouldTerminateAfterLastWindowClosedという、「最後のウィンドウが閉じられた時に アプリごと終了してよいか」をAppKitがアプリ側に尋ねてくるデリゲートメソッドです。これが (実装していなかったため既定で)trueを返す扱いになっていたため、何らかの拍子で パネルが「閉じた」扱いになった瞬間、AppKitの内部処理がタイマー経由でアプリ全体を終了させていました。 

func applicationShouldTerminateAfterLastWindowClosed(_ sender: NSApplication) -> Bool {
    // このアプリはメニューバー常駐のユーティリティなので、
    // パネルが(一瞬でも)閉じた扱いになってもアプリ全体は終了させない。
    // 終了はメニューバーの「終了」からのみ行う。
    false
}

メニューバー常駐アプリ(ステータスバーアイコンから操作するタイプ)では、 「メインウィンドウを閉じる」と「アプリを終了する」を分離するのが定石です。この1行を直すことで、 パネルを閉じてもアプリはメニューバーに残り、「パネルを表示」でいつでも復帰できるようになりました。

5. バグ4(最大の難所): ウィンドウが画面いっぱいまで巨大化する

UIを整えている途中で、ウィンドウが指定した420×280を大きく超えて、 縦に1000ptを超えるサイズまで膨れ上がる現象が発生しました。SwiftUI側のレイアウト(Spacerや .background()など)を1つずつ疑って除去しても直らず、原因の特定に非常に時間がかかりました。

ハンズオン: 二分探索でバグを追い詰める

原因が分からないコードの中から犯人を1つずつ見ていくのは非効率です。こういうときに有効なのが 二分探索的なアプローチです。「疑わしい範囲を半分に割って、どちら側に原因があるかを 確認する」を繰り返すことで、少ない試行回数で原因を絞り込めます。今回は、複雑なSwiftUIの 画面をいきなり全部調べるのではなく、まず画面の中身を極限まで単純化した最小の例に 差し替えて動作を見る、という形で、この考え方を応用しました。 

let hosting = NSHostingView(rootView: Text("Hello TEMP DEBUG"))

このとき、ウィンドウは指定した420×280ではなく、Textの内容にぴったり合った 119×48という小さいサイズになりました。「複雑なUIだと巨大化し、単純なUIだと逆に縮む」—— この2つの実験結果を見比べることで、SwiftUIコンテンツの理想サイズ(fitting size。 「このくらいの大きさで表示したい」というSwiftUI側の希望サイズ)がウィンドウそのものを 書き換えているという仮説にたどり着きました。「怪しい箇所を1つずつ削って、最小の状態でも 症状が再現するかを確認する」というこの手順は、原因不明のバグ全般に応用できる、 汎用性の高いデバッグの型です。

実際、NSHostingViewは既定でsizingOptionsというプロパティを持っており、 SwiftUI側のレイアウト結果(intrinsic content size)を自分の親ウィンドウのサイズにまで伝播させます。 通常のSwiftUI Appでは「ウィンドウの初期サイズをコンテンツに合わせたい」場面で便利な機能ですが、 今回のように「ウィンドウサイズは自分たちで完全に管理したい」場合には邪魔になります。 

let hosting = NSHostingView(rootView: ContentView(viewModel: viewModel))
// NSHostingViewは既定でSwiftUI側の理想サイズ(fittingSize)に合わせて
// ウィンドウ自体をリサイズしてしまう。ウィンドウサイズは自分たちで管理したいので無効化する
hosting.sizingOptions = []

この1行を追加した瞬間、何十行ものレイアウト修正を試しても直らなかった巨大化が完全に解消しました。

Tips: frameの実測ログで「思い込み」を排除する

このバグ調査で決定的だったのは、スクリーンショットの目視ではなく、 DispatchQueue.main.asyncAfterで複数のタイミングにおけるpanel.frameと hosting.fittingSizeを実際にprintしたことでした。 

for delay in [0.0, 0.5, 1.0, 2.0, 5.0] {
    DispatchQueue.main.asyncAfter(deadline: .now() + delay) {
        print("t=\(delay) window.frame=\(newPanel.frame) " +
              "hosting.fittingSize=\(hosting.fittingSize)")
    }
}

スクリーンショットの目測は「他のウィンドウとの重なり」「Retinaのスケール換算」などでいくらでも 誤読が起きます。数値を直接ログに出すことで、「本当にウィンドウのframeが変わっているのか」 「それとも見た目だけの問題か」を一発で切り分けられました。

6. ウィンドウレベルを状況に応じて切り替える

「常に最前面」は便利ですが、他のアプリを操作しているときまでFloatPlayerが手前に居座るのは 邪魔になります。そこでNSWorkspace(実行中のアプリの一覧やアクティブなアプリなど、 macOS全体の状態を扱うためのクラス)が発行する「アプリ切り替え通知」を購読し、自分が アクティブな間だけ最前面、それ以外は通常レベルに切り替えるようにしました。 NotificationCenter(通知センター)は、Combineの@Published/sink (Swift基礎編2章)よりも古くからAppKitにある、 同じ「何かが起きたら知らせる」仕組みです。考え方はCombineとよく似ていますが、より広い範囲 (システム全体のイベントなど)に向けて使われます。 

NSWorkspace.shared.notificationCenter.addObserver(
    self,
    selector: #selector(handleAppActivation(_:)),
    name: NSWorkspace.didActivateApplicationNotification,
    object: nil
)

@objc private func handleAppActivation(_ note: Notification) {
    guard let app = note.userInfo?[NSWorkspace.applicationUserInfoKey]
        as? NSRunningApplication else { return }
    panel.level = app.processIdentifier == ProcessInfo.processInfo.processIdentifier
        ? .floating : .normal
}

NSWorkspace.didActivateApplicationNotificationは、自分を含むどのアプリがアクティブになっても 発火します。userInfoからNSRunningApplication(今アクティブになったアプリの情報)を 取り出し、そのプロセスID(OSが実行中のプログラム(プロセス)それぞれに割り振る、重複しない 識別番号)が自分自身のもの(ProcessInfo.processInfo.processIdentifier)かどうかで、 「今アクティブなのは自分か、他のアプリか」を判定しています。

7. 機能: スクリーンショットの自動フローティングウィンドウ

Swift基礎編で扱ったScreenshotWatcherが 新しいスクリーンショットを検知すると、ScreenshotFloatWindowが「UIなし・画像だけ」の 小さなフローティングウィンドウを画面に浮かべます。メインのプレイヤーパネルとは別に、 もう1つNSPanelを使う実装なので、設定内容を比較しながら見ていきます。 

let newPanel = NSPanel(
    contentRect: NSRect(origin: Self.nextSpawnPoint(), size: size),
    // .titled+.resizableは1章のメインパネルと同じ組み合わせ。
    // タイトルバーを透明・非表示にすることで見た目はボーダーレスにしている
    // (なぜ素直に.borderlessにしなかったかは次の8章で扱う)
    styleMask: [.titled, .resizable, .nonactivatingPanel, .fullSizeContentView],
    backing: .buffered,
    defer: false
)
newPanel.titleVisibility = .hidden
newPanel.titlebarAppearsTransparent = true
newPanel.standardWindowButton(.closeButton)?.isHidden = true
newPanel.standardWindowButton(.miniaturizeButton)?.isHidden = true
newPanel.standardWindowButton(.zoomButton)?.isHidden = true
newPanel.isOpaque = false
newPanel.backgroundColor = .clear
newPanel.hasShadow = true
newPanel.level = .floating
newPanel.collectionBehavior = [.canJoinAllSpaces, .fullScreenAuxiliary]
設定意味
.titled + titleVisibility = .hiddenタイトルバーの「実体」は残しつつ見た目だけ消す。実体を残す理由は8章参照
standardWindowButton(...)?.isHidden信号機ボタンを個別に非表示にする(styleMaskに.closable等を含めないだけだとグレーアウト表示のまま残る)
.canJoinAllSpacesMission Controlのどのデスクトップ(Space)に切り替えても付いてくる

画像のドラッグ移動は、NSImageViewではなく自作の軽量なNSViewサブクラスで実現しています。 NSImageViewは既定で「画像を他アプリへドラッグ書き出しする」機能を持っており、これが mouseDownを横取りしてしまうため、isMovableByWindowBackgroundによるウィンドウの 背景ドラッグが効かなくなります。レイヤーに画像を描画するだけの自作ビューに置き換え、 mouseDownで直接performDrag(with:)を呼ぶことで確実に動かせるようにしています。 

override func mouseDown(with event: NSEvent) {
    window?.performDrag(with: event)
}

右クリックメニュー(閉じる/保存/コピー)は、自作ビューにNSMenu(メニューの中身を表すクラス)を 直接持たせるだけで実現できます。SwiftUIのコンテキストメニューという仕組みを介さなくても、AppKitのビューには ネイティブのメニューをそのまま生やせる、というのがAppKitならではの手軽さです。 透明度調整のスライダーは、右クリックメニューのNSMenuItem.view(メニュー項目1つ1つに、 ボタンの代わりに好きなビューを差し込めるプロパティ)にカスタムビュー(NSSlider入り)を 差し込むことで、メニューの中に埋め込んでいます。 

let menu = NSMenu()
let opacityItem = NSMenuItem()
opacityItem.view = OpacityMenuItemView(initialValue: newPanel.alphaValue) { value in
    newPanel.alphaValue = value
}
menu.addItem(opacityItem)
menu.addItem(withTitle: "閉じる", action: #selector(closeSelf), keyEquivalent: "").target = self
imageView.menu = menu
Tips: 連続スクリーンショットが重ならないよう、出現位置をずらす

立て続けにスクリーンショットを撮ると、フローティングウィンドウが毎回まったく同じ位置に出て すべて重なってしまいます。呼び出し回数を数え、6パターンの位置を斜めにずらして順番に割り当てることで、 複数枚を同時に見比べられるようにしています。 

private static var lastSpawnIndex = 0
private static func nextSpawnPoint() -> NSPoint {
    defer { lastSpawnIndex += 1 }
    let offset = CGFloat((lastSpawnIndex % 6) * 32)
    return NSPoint(x: 220 + offset, y: 420 - offset)
}

deferは「この関数を抜ける直前に、必ずこの処理を実行する」という予約です。 ここでは「戻り値の計算に使う前の値のまま位置を計算しつつ、関数を抜けるタイミングで 確実にインクリメント(1増やす)する」ために使っています。

8. バグ5(最難関): .borderlessウィンドウではカーソル形状が変えられない

スクリーンショットのフローティングウィンドウを端からドラッグして拡大縮小できるようにしたとき、 「端にカーソルを合わせるとカーソルの形が矢印⇔リサイズ用に変わる」という、ブラウザなどでは 当たり前の挙動を再現しようとして、6回連続で失敗しました。最終的な原因は実装の細部ではなく、 .borderlessウィンドウという選択そのものにありました。

実際に踏んだバグ: 6つの方式すべてがカーソルを変えられなかった

試した順に並べると次の通り。

1. resetCursorRects() + addCursorRect(...)
   → 反映されるときとされないときがあり、判定できない
2. NSTrackingArea + cursorUpdate(with:)
   → 自分のアプリがアクティブな間は動くが、他アプリ(ブラウザ等)が
     アクティブな間は呼ばれない
3. NSEvent.addLocalMonitorForEvents + addGlobalMonitorForEvents
   → 他アプリの環境と競合し、反応する時としない時がある
4. Timer.scheduledTimerでの定期ポーリング + NSCursor.set()
   → AppKit標準の「cursor rectが無ければ矢印に戻す」処理と競合し、
     ほぼ勝てない
5. resetCursorRects()を残しつつstyleMaskから.resizableだけ外す
   → 変化なし
6. 上記の組み合わせ(resetCursorRects + 非アクティブ時だけタイマー)
   → それでも変わらない

6回目の失敗を受けて発想を変え、「メインパネルは.titled+.resizableで 問題なくリサイズもカーソルも機能している」という、すでに動いている実装と見比えることにしました。 結論は単純で、NSCursor.set()やresetCursorRects()によるカーソル制御は、 .titledのような本物のウィンドウ枠(border)を持つウィンドウでないと安定して機能しません ——.borderlessウィンドウでは、標準のカーソル制御機構そのものに乗れていませんでした。

そこで、リサイズを自前のmouseDown/mouseDragged判定で実装するのをやめ、7章の通り .titled + .resizableにタイトルバーを透明・非表示にする方式へ切り替えました。 すると、自前のコードを1行も書かずに、AppKit標準のedge-dragリサイズとカーソル形状の変更が そのまま正しく動くようになりました。

教訓: 「動いている実装との差分」から疑う

6回の試行はすべて「カーソル制御の実装方法」を変える方向の修正で、いずれも症状の一部にしか 効きませんでした。実際の原因は実装方法ではなくstyleMaskの選択(.borderless vs .titled)という、 もっと手前の前提にありました。原因不明のバグが同じ機能の別実装(この場合はメインパネル)で 起きていないなら、両者の設定の差分を突き合わせるほうが、個別の対症療法を繰り返すより早い。

なお、.titled+.resizableにしたことで、縦横比を保ったまま拡大縮小する NSWindow.contentAspectRatioも使えるようになりました。ただし今回は「画像を自由に伸び縮みさせたい」 という要望だったため、あえて設定していません。画像側もCALayer.contentsGravityを 縦横比を保つ.resizeAspectではなく、枠に合わせて引き伸ばす.resizeにすることで、 ウィンドウの変形に画像そのものが追従するようにしています。
